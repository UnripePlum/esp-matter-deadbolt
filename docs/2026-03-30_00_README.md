## 2. 하드웨어 스펙

### 2-1. 릴레이 (CW-020 1채널)

| 항목 | 값 |
|------|-----|
| 릴레이 | CW-020 1채널 (Low Level Trigger) |
| GPIO | 4 (IN 핀) |
| 배선 | COM → EXIT 1, NO → EXIT 2 |
| GPIO LOW | 릴레이 OFF → 잠금 (부팅 기본값) |
| GPIO HIGH | 릴레이 ON → 해제 (지속) |

### 2-2. GPIO 할당

| GPIO | 기능 | 방향 | 설명 |
|------|------|------|------|
| 0 | BOOT 버튼 | 입력 (풀업) | 5초 길게 누르기 → 팩토리 리셋 |
| 4 | CW-020 릴레이 IN | 출력 | LOW=잠금, HIGH=해제 |
| 48 | 상태 LED (WS2812) | 출력 | led_driver 사용 |

## 3. Matter API (Door Lock Cluster 0x0101)

WiFi over Matter 통신. chip-tool 또는 Matter 컨트롤러에서 사용.

### 3-1. 명령 (Commands)

| 명령 | CHIP 콜백 | 동작 |
|------|-----------|------|
| `LockDoor` | `emberAfPluginDoorLockOnDoorLockCommand` | `door_queue_command(false)` → GPIO LOW (잠금) |
| `UnlockDoor` | `emberAfPluginDoorLockOnDoorUnlockCommand` | `door_queue_command(true)` → GPIO HIGH (해제) |
| `UnlockWithTimeout` | `emberAfPluginDoorLockOnDoorUnlockWithTimeoutCommand` | `door_exit_open(timeout)` → GPIO HIGH → 타이머 후 자동 잠금 |

### 3-2. 속성 (Attributes)

| 속성 | ID | 타입 | 값 |
|------|----|------|-----|
| LockState | 0x0000 | enum8 | 0=NotFullyLocked, 1=Locked, 2=Unlocked |
| LockType | 0x0001 | enum8 | 0 (DeadBolt) |
| ActuatorEnabled | 0x0002 | bool | true |

### 3-3. 이벤트 처리

| 이벤트 | 동작 |
|--------|------|
| `kCommissioningComplete` | `comm_set_matter_connected(true)` + LED 갱신 |
| `kInterfaceIpAddressChanged` (IPv4) | `comm_set_matter_connected(true)` |
| `kWiFiConnectivityChange` (Lost) | `comm_set_matter_connected(false)` |
| `kCommissioningWindowOpened/Closed` | LED 커미셔닝 표시 갱신 |
| `kFabricCommitted` / `kServerReady` | LED 커미셔닝 표시 갱신 |

### 3-4. Attribute Update Callback

`app_attribute_update_cb`는 `LockState` 변경 시 **로그만 출력**. 릴레이 조작은 `emberAf` 콜백에서만 처리 (피드백 루프 방지).

### 3-5. 미지원 기능

User, Credential, Schedule 관련 콜백은 모두 `false` / `kFailure` 반환. PIN 코드, 사용자 관리, 스케줄은 지원하지 않음.

## 4. BLE API (GATT 폴백 채널)

Matter 연결 불가 시 BLE GATT 서버를 통한 폴백 제어.

### 4-1. 서비스 정보

| 항목 | 값 |
|------|-----|
| Service UUID | `0000FF01-0000-1000-8000-00805F9B34FB` |
| Device Name | `MatterDoorLock` |
| 보안 | LE Secure Connections + MITM + Bonding |
| 암호화 | AES-128-GCM (IV: counter 4B + random 8B = 12B) |
| 인증 | Challenge-Response (HMAC-SHA256) |

### 4-2. Characteristics

| Characteristic | UUID | 속성 | 용도 |
|---------------|------|------|------|
| Command | FF02 | Write (암호화 필수) | 명령 수신 |
| Status | FF03 | Read, Notify (암호화 필수) | 상태/결과 전송 (AES-GCM) |
| Config | FF04 | Read, Write (암호화 필수) | 퇴실 기본시간 설정 (3~30초) |
| Challenge | FF05 | Read (암호화 필수) | 일회용 nonce 발급 (16B) |

### 4-3. Command 바이트 정의

| 명령 | 코드 | 인증 | 파라미터 | 동작 |
|------|------|------|----------|------|
| LOCK | `0x01` | 불필요 | 무시 | `door_queue_command(false)` |
| UNLOCK | `0x02` | **HMAC 필수** | 무시 | `door_queue_command(true)` |
| EXIT_OPEN | `0x03` | **HMAC 필수** | duration (초) | `door_exit_open(param)` |
| STATUS_REQ | `0x04` | 불필요 | 무시 | `ble_send_current_status()` |

### 4-4. Status 바이트 정의

| 상태 | 코드 | 설명 |
|------|------|------|
| LOCKED | `0x01` | 잠금 상태 |
| UNLOCKED | `0x02` | 해제 상태 |
| OP_SUCCESS | `0x10` | 동작 성공 |
| OP_FAIL | `0x11` | 동작 실패 |
| AUTH_FAIL | `0x12` | HMAC 인증 실패 |
| ERROR | `0xFF` | 일반 에러 |

### 4-5. Challenge-Response 프로토콜

UNLOCK(`0x02`)과 EXIT_OPEN(`0x03`) 명령에 필수.

```
┌─ Client ─────────────────────── Server ──┐
│ 1. Read Challenge(FF05)  →  nonce(16B)   │
│ 2. HMAC = SHA256(AES_KEY, nonce||cmd||p) │
│ 3. Write Command(FF02): [cmd,p,HMAC(32)] │
│ 4.                    ←  검증 후 실행     │
└──────────────────────────────────────────┘
```

**패킷 구조:** `[cmd(1B), param(1B), HMAC(32B)]` = 34바이트

**보안 특성:**
- 일회용 nonce (사용 후 즉시 무효화)
- 30초 TTL (`BLE_CHALLENGE_TTL_MS`)
- BLE 연결 해제 시 자동 무효화
- Constant-time HMAC 비교 (타이밍 공격 방지)

### 4-6. AES-GCM 암호화 (Status Notify)

Status Notify는 항상 AES-128-GCM으로 암호화되어 전송.

**패킷 구조:** `[IV(12B), Ciphertext(len), Tag(16B)]`

| 구성요소 | 크기 | 설명 |
|----------|------|------|
| IV | 12B | counter(4B) + random(8B) |
| Ciphertext | 최대 16B | 암호화된 상태 데이터 |
| Tag | 16B | GCM 인증 태그 |

**키 관리:**
- 부팅 시 NVS `ble_crypto/aes_key`에서 로드
- 미발견 시 `esp_random()`으로 16바이트 생성 후 NVS 저장
- Nonce counter: NVS 영속, 부팅 시 +100 점프, 100회마다 NVS 저장

## 5. 도어 컨트롤러 (door_controller)

### 5-1. Public API

| 함수 | 설명 |
|------|------|
| `door_controller_init()` | 초기화 (mutex, 타이머, 큐, 워커 태스크) |
| `door_execute(bool target_unlock)` | 직접 실행 (mutex 3초 타임아웃, `OP_RESULT_FAIL_BUSY` 반환 가능) |
| `door_queue_command(bool target_unlock)` | 큐에 등록 (크기 1, `xQueueOverwrite`로 최신 명령 덮어쓰기) |
| `door_exit_open(uint8_t duration_sec)` | 퇴실: 해제 → duration_sec 후 자동 잠금 (3~30초, 기본 5초) |
| `door_is_locked()` | 소프트웨어 잠금 상태 반환 |

### 5-2. 동작 결과

| 결과 | 값 | 설명 |
|------|----|------|
| `OP_RESULT_SUCCESS` | 0 | 성공 |
| `OP_RESULT_FAIL_BUSY` | 1 | 다른 동작 진행 중 (mutex 타임아웃) |

### 5-3. 동작 흐름

```
door_queue_command(unlock=true)
  → xQueueOverwrite (크기 1, 최신 명령만)
  → door_worker_task가 수신
  → door_execute(true)
     → xSemaphoreTakeRecursive (3초 타임아웃)
     → deadbolt_unlock() — GPIO 4 HIGH
     → g_lock_state = false
     → report_result(SUCCESS, 1)
     → status_led_set_locked(false)
     → xSemaphoreGiveRecursive
```

### 5-4. 퇴실 기능 (EXIT_OPEN)

```
door_exit_open(5)
  → door_execute(true) — 즉시 해제
  → esp_timer_start_once(5초)
  → ... 5초 후 ...
  → exit_timer_cb → xTaskCreate(exit_lock_task)
  → exit_lock_task → door_execute(false) — 자동 잠금
```

### 5-5. 설정값

| 상수 | 값 | 설명 |
|------|----|------|
| `DEFAULT_EXIT_DURATION_SEC` | 5 | 퇴실 기본 열림 시간 |
| `EXIT_MIN_SEC` | 3 | 퇴실 최소 시간 |
| `EXIT_MAX_SEC` | 30 | 퇴실 최대 시간 |

```
  └─ 둘 다 불가? → 로그 경고 (NVS 큐 제거됨)
```

## 8. Status LED (WS2812)

### 8-1. Public API

| 함수 | 설명 |
|------|------|
| `status_led_init()` | WS2812 초기화 + LED 태스크 시작 |
| `status_led_set_system_ready(bool)` | 부팅 완료 표시 |
| `status_led_set_commissioning(bool)` | 커미셔닝 대기 표시 |
| `status_led_set_locked(bool)` | 잠금/해제 상태 표시 |
| `status_led_notify_locking()` | 잠금 진행 피드백 (2초) |
| `status_led_notify_unlocking()` | 해제 진행 피드백 (2초) |
| `status_led_notify_op_fail()` | 동작 실패 피드백 (2초) |
| `status_led_get_state_str()` | 현재 LED 상태 문자열 |

### 8-2. 시각 상태

| 상태 | 색상 | 패턴 | 우선순위 |
|------|------|------|----------|
| 부팅 중 | 파랑 | 느린 깜빡임 (500ms) | 2 (시스템) |
| 커미셔닝 대기 | 파랑 | 빠른 깜빡임 (200ms) | 2 (시스템) |
| 잠금 완료 | 주황 | 고정 | 3 (일반) |
| 해제 완료 | 초록 | 고정 | 3 (일반) |
| 잠금 진행 중 | 주황 | 빠른 깜빡임 (150ms) | 1 (피드백) |
| 해제 진행 중 | 초록 | 빠른 깜빡임 (150ms) | 1 (피드백) |
| 동작 실패 | 빨강 | 빠른 깜빡임 (150ms) | 1 (피드백) |

우선순위: 피드백(1) > 시스템(2) > 일반(3). 피드백은 2초 후 자동 해제.

## 9. 초기화 순서 (app_main)

```
1. GPIO 4 즉시 LOW 고정 (잠금 — 부팅 시 안정화)
2. gpio_init() — GPIO 전체 초기화
3. NVS Flash 초기화
4. ble_crypto_init() — AES 키 + Nonce counter 로드 (NimBLE 불필요)
5. Matter 노드 생성 (attribute_cb, identification_cb)
6. Door Lock 엔드포인트 생성
7. door_controller_init() — mutex, 큐, 워커 태스크, 타이머
8. esp_matter::start() — Matter 시작 (NimBLE 초기화 포함)
9. ble_gatt_server_init() — GATT 서비스 등록 (Matter NimBLE 위에)
10. status_led_init() + 시스템 준비 완료
11. factory_reset_task 시작 (BOOT 버튼 5초 모니터링)
```

**핵심:** `ble_crypto_init()`은 NVS만 사용하므로 Matter 시작 전 호출. `ble_gatt_server_init()`은 NimBLE가 필요하므로 반드시 `esp_matter::start()` 이후 호출.

## 10. 테스트 (deadbolt_api_test REPL)

```bash
./deadbolt_api_test [node_id] [endpoint_id]
```

### 10-1. 도어락 제어

| 명령 | 동작 |
|------|------|
| `/lock` | Matter LockDoor → GPIO LOW (잠금) |
| `/unlock` | Matter UnlockDoor → GPIO HIGH (해제) |
| `/exit_open [초]` | Matter UnlockWithTimeout → 퇴실 열림 (기본 5초, 자동 잠금) |

### 10-2. 상태 읽기

| 명령 | 동작 |
|------|------|
| `/state` 또는 `/status` | LockState 읽기 (1=Locked, 2=Unlocked) |

### 10-3. OTA 업데이트

| 명령 | 동작 |
|------|------|
| `/ota [bin] [ver]` | OTA 이미지 생성 → Provider 시작 → 커미셔닝 → ESP32에 트리거 |
| `/ota_stop` | OTA Provider 종료 + 커미셔닝 해제 |

**OTA 절차:**
1. `ota_image_tool.py`로 `.ota` 이미지 생성 (VendorID=0xFFF1, ProductID=0x8000)
2. `chip-ota-provider-app` 백그라운드 실행
3. Provider를 node 2로 커미셔닝
4. `otasoftwareupdaterequestor announce-otaprovider`로 ESP32에 트리거

### 10-4. 커미셔닝

| 명령 | 동작 |
|------|------|
| `/pair [pin]` | 온네트워크 커미셔닝 (기본 pin: 20202021) |
| `/pair_wifi <ssid> <pw>` | BLE WiFi 커미셔닝 |
| `/pair_auto` | 저장된 WiFi 크레덴셜로 자동 BLE 커미셔닝 |
| `/unpair` | 커미셔닝 해제 |

### 10-5. 기타

| 명령 | 동작 |
|------|------|
| `/smoke` | 스모크 테스트 (상태 → 잠금 → 확인 → 해제 → 확인) |
| `/node` | 현재 node/endpoint 확인 |
| `/help` | 도움말 |
| `/exit` | REPL 종료 |
| `chip-tool ...` | 아무 chip-tool 명령 직접 실행 |
